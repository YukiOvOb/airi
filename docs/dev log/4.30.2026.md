# DevLog @ 2026.04.30 - 记忆工具暴露 & TTS 数据源重构

## 概述

本次更新围绕两个独立但都长期被忽视的问题：

1. **记忆系统的"读写不对称"**——AI 看得到记忆，却写不进新记忆，因为根本没有把写入操作暴露成 LLM 可调用的工具
2. **TTS 与 UI 对话框的数据源不一致**——两条并行管线各自处理 LLM 输出，听感和视觉时常对不上号

最终一并解决。同时为 TTS 加了"延迟打断"逻辑：用户发送新消息时旧 TTS 不立即停下，而是等到 AI 真正开口回答的那一刻才切换。

---

## 一、记忆系统：从只读到可写

### 背景

之前的记忆系统是单向的：

```
存储层 (memoryStore.upsertMemory)  ← UI 设置页能调
        ↓
读取层 (memory context provider)   ← 注入到系统提示词
        ↓
       LLM
```

LLM 看到 `<memory>` 标签里的内容，但**没有任何工具能把新观察写回去**。结果 AI 凭直觉到处乱猜工具名：

- `airi:system:memory:update` —— 把上下文 provider 的 plugin id 当工具名
- `memory.writeObject` —— OpenAI 风格的 dot-namespaced 命名
- `memory.write*` —— 各种变体

每次都被 `@xsai/shared-chat` 抛出 `Model tried to call unavailable tool` 直接拒绝，记忆永远只有人工录入的几条。

### 三层修复

#### 1. 创建工具并注册到 LLM 工具列表

新建 `packages/stage-ui/src/tools/memory.ts`，定义三个工具：

```ts
builtIn_memoryUpsert // 创建/更新记忆，参数对齐 memoryStore.upsertMemory
builtIn_memoryDelete // 按 id 删除
builtIn_memoryList // 列出当前记忆，可按 type / minImportance 过滤
```

在 `packages/stage-ui/src/stores/llm.ts` 的工具组装处加上 `...await memory()`，与原本就存在但被禁用的 `mcp()`、`debug()`、`createSparkCommandTool` 并列。

#### 2. 在系统提示词里写死工具名

光定义工具不够——LLM 看到 `<memory>` 块仍然会按训练数据里见过的命名风格自由发挥。修改 `packages/stage-ui/src/stores/chat/context-providers/memory.ts`，在 `<memory>` 块尾部强制注入工具目录：

```markdown
## 记忆管理工具

如需新增、更新或删除记忆，**只能**调用以下工具（其它名称都不存在，调用会报错）：

- `builtIn_memoryUpsert` — 创建或更新一条记忆
- `builtIn_memoryDelete` — 按 `id` 删除一条记忆
- `builtIn_memoryList` — 列出当前记忆，可按 `type` 或 `minImportance` 过滤

不要尝试调用 `memory.write*`、`airi:system:memory:*` 或任何其它名称的记忆工具。
```

并把"无记忆时返回 null"改成"无记忆时也注入工具目录"，这样新角色第一次对话就能保存第一条记忆。

#### 3. OpenAI 严格模式 schema 兼容

第一次跑起来还是报错：

```
Invalid schema for function 'builtIn_memoryUpsert': In context=(), 'required'
is required to be supplied and to be an array including every key in
properties. Missing 'description'.
```

OpenAI 的 strict-mode tool schema 要求 `properties` 中**每一个键**都必须出现在 `required` 数组里。zod 的 `.optional()` 会让字段从 `required` 中移除，违反规则。

按项目里 `tools/character/orchestrator/spark-command-shared.ts` 已有的模式重写——所有字段都是必填，"可选"语义改用 `null`：

```ts
// 之前
description: z.string().optional()

// 之后
description: z.union([z.string(), z.null()])
```

并给所有 object schema 加 `.strict()`。Runtime 层用 `?? undefined` 转回项目默认值。

## 二、TTS 数据源：从流式 delta 切到 UI 对话框文字

### 旧架构问题

```
LLM 流式 delta ─┬─→ 分类器 → 字面量解析器 → TTS 队列
                └─→ 分类器 → buildingMessage.content → UI 对话框
```

两条管线各自跑分类器、各自处理特殊 token、各自喂下游。理论上结果一致，实际上 `Stage.vue:onTokenLiteral` 还会再调一次 `categorizeResponse(literal)`（双重过滤），任何上下游不一致就会出现"看到的 ≠ 听到的"。

用户的诉求：**TTS 单一数据源 = UI 对话框文字**。

### 整段读取方案

最初实现了增量 watch 版本（监听 `streamingMessage.value.content` 的增长，把 delta 推给 TTS），保留了"边读边说"的体验。但用户最终选了更简单的整段读取：

```
LLM 流式 delta → categorizer → buildingMessage.content → UI 对话框
                                          ↓ (响应式)
                                streamingMessage.value
                                          ↓
                              onAssistantResponseEnd
                                          ↓
                                 currentChatIntent
                                  .writeLiteral(...)
                                  .end()
```

关键改动：
- **删掉** `Stage.vue` 中原有的 `onTokenLiteral` TTS 喂入钩子（含 `categorizeResponse` 二次过滤）
- **删掉** `categorizeResponse` 在 `Stage.vue` 的 import
- **新增** 在 `onAssistantResponseEnd` 一次性写入 `streamingMessage.value.content`
- **保留** `onTokenSpecial`（emotion token 仍即时触发角色表情指令）

```ts
chatHookCleanups.push(onAssistantResponseEnd(async (_message) => {
  characterRuntimeStore.onAssistantResponseEnd(_message)
  const bubbleText = streamingMessage.value.content
  if (currentChatIntent && typeof bubbleText === 'string' && bubbleText.length > 0)
    currentChatIntent.writeLiteral(bubbleText)
  currentChatIntent?.end()
  currentChatIntent = null
}))
```

### 取舍

| 方案 | 体验 | 复杂度 |
|---|---|---|
| 增量 watch | 边显示边说，TTS 即时跟随 | 需游标 + diff + shrink 检测 |
| **整段读取** | 等消息显示完才开口 | 一行 writeLiteral |

最终选了整段方案：实现简单、行为可预测，且因为 LLM 流式速度通常远快于 TTS 合成，"等消息显示完"也就几秒之内的事，体感差异不大。

## 三、双路打断：语音打断 + 文字打断的延迟触发

整段读取模式打开了一个新窗口期：用户发送新消息时，新 intent 已经打开但还**空着**（要等 `onAssistantResponseEnd` 才喂文字）。这要求重新审视打断逻辑。

### 语音打断补漏

原来的实现：

```ts
onSpeechDetected: () => {
  playbackManager.stopAll('user-interrupt')
  speechPipeline.interrupt('user-interrupt')
}
```

`speechPipeline.interrupt(reason)` 看 `speech-pipeline.ts:354-357` 只取消 **active** intent。整段读取模式下，AI 流式期间我们的 `currentChatIntent` 经常不是 active（因为还空着），就会被这次打断"漏掉"，等流式结束 `onAssistantResponseEnd` 触发 `writeLiteral` 时仍然会被读出来——跟用户的"打断"意图相反。

补上：

```ts
onSpeechDetected: () => {
  if (interruptEnabled.value) {
    playbackManager.stopAll('user-interrupt')
    speechPipeline.interrupt('user-interrupt')
    if (currentChatIntent) {
      currentChatIntent.cancel('user-interrupt')
      currentChatIntent = null
    }
    if (previousChatIntent) {
      previousChatIntent.cancel('user-interrupt')
      previousChatIntent = null
    }
    pendingResponseStart = false
  }
}
```

`intent.cancel()` 是幂等的（`speech-pipeline.ts:336-339` 通过 `if (!intent) return` 保证二次调用 no-op），所以即便和 `pipeline.interrupt` 重叠取消同一个 intent 也安全。

### 文字打断从"立即"改为"延迟到 AI 开口"

#### 旧行为

```
用户发送 → onBeforeMessageComposed
  · playbackManager.stopAll('new-message')   ← 立即停旧 TTS
  · currentChatIntent.cancel('new-message')   ← 取消旧 intent
  · 开新 intent
```

旧 TTS 戛然而止，到 AI 开始回复之间是一段静默。

#### 新行为

```
用户发送 → onBeforeMessageComposed
  · 把旧 intent 暂存到 previousChatIntent
  · pendingResponseStart = true
  · 开新 intent（queue 模式，自动排在旧 intent 之后）
  · 不动 playbackManager

AI 开始回答 → 首个 literal token → onTokenLiteral (新增钩子)
  · pendingResponseStart 翻转为 false（一次性触发）
  · playbackManager.stopAll('new-message')
  · cancel previousChatIntent
  · pipeline 的 runIntent finally 块自动 pickNext，新 intent 升为 active
```

#### 状态机

新增两个顶层变量：

```ts
let previousChatIntent: ... | null = null  // 暂存上一轮 intent，等首 token 才 cancel
let pendingResponseStart = false           // 标记"已发送但 AI 未开口"
```

边界情况：
- **思考型模型**先输出 `<think>...</think>` 被分类器过滤，不算 literal token；这段时间旧 TTS 继续播——符合"AI 还没真正开口"的语义
- **用户连发两条**：第二次 `onBeforeMessageComposed` 进来时 `previousChatIntent` 已被占用，先 cancel 它再放新值，避免 intent 链堆积
- **中间发生语音打断**：把 `currentChatIntent`、`previousChatIntent` 全部 cancel 并复位 `pendingResponseStart`

#### 选用 onTokenLiteral 作为信号的理由

`onTokenLiteral` 在 `chat.ts:382-385` 中只在分类器过滤后的"speech-only"文本到达时才 fire——精确等价于"AI 输出了一个会显示在对话框里的字"。这正是用户想要的"AI 开始回答"时刻。

emotion token (`onTokenSpecial`) 不算，因为它们不显示在对话框里。

## 时序对比图

### 旧 TTS 单路 + 立即打断

```
t0  旧 TTS 在播 ───────────────╮
t1  用户发送                    │ 立即停
                                ▽
t2  static silence ────────────╮
t3  AI 流式回复 ────────────────│
t4  TTS 边流边播 ───────────────╯
```

### 新 TTS 整段读取 + 延迟打断

```
t0  旧 TTS 在播 ─────────────────────────╮
t1  用户发送 (intent 入队等待)             │
t2  AI 思考中 (<think> 被过滤)            │
t3  AI 首个 literal token ──────────────╮│
                                         ▽▽
t4  AI 流式继续 → buildingMessage 累积    │
t5  AI 流式结束 → writeLiteral(整段)      │
t6  新 TTS 一次性合成播放 ─────────────────╯
```

后者的核心好处是：旧 TTS 完整播完用户已能听清的最后那段话，没有突兀的截断。

## 后续可考虑

### Captions 同步延迟

目前 `assistantCaption` / `postCaption` 在 `onBeforeMessageComposed` 立即清空。理论上应该和 TTS 一起延迟到 `onTokenLiteral` 首次触发——否则用户发送后会出现"字幕清空但音频还在响"的短暂错位。

没立刻改是因为这跟当前任务焦点（TTS）正交，且字幕错位通常只是几百毫秒，影响有限。如果需要再加一个 `pendingCaptionReset` 标志即可。

### Emotion token 与整段 TTS 的对齐

整段读取模式下：
- emotion token 在流式期间通过 `writeSpecial` 立即写入 intent 队列
- 整段文字在 `onAssistantResponseEnd` 一次性写入

intent 内部按写入顺序处理，所以最终顺序是：emotion 先、文字后。如果 emotion 是基于位置的（例如"在第三句话之后转头"），就会失去时间对齐。当前实现里 emotion 主要驱动角色表情切换（即时反应），所以这个错位影响不大。

未来如果要支持"句中触发的精细化表情"，可能需要给 intent 加"重排序到正确位置"的能力，或者让 emotion 也跟整段一起写入。

### 工具命名规范

`builtIn_memoryUpsert` 这套命名跟项目里 `airi:system:*`、`character.*` 等命名风格不太一致。命名上的统一暂时没做，保留为 followup——主要因为现有 LLM 已经习惯了 OpenAI 风格的 underscore_camelCase。
