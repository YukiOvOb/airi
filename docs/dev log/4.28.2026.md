# DevLog @ 2026.04.28 - Embedding-Based Memory Retrieval

## 概述

实现了基于 Embedding 的记忆相关性召回功能，让 AIRI 能够根据用户当前的对话内容，智能地检索相关的长期记忆，而不是简单地将所有记忆注入到上下文中。

## 背景

之前的记忆系统采用"全量注入"策略：
- 当记忆数量少于 25KB/200 行时，将所有记忆注入系统提示
- 当记忆超过限制时，只注入 MEMORY.md 索引

这导致两个问题：
1. **记忆多了之后体验退化**——AI 只能看到索引，看不到具体内容
2. **没有按相关性检索**——即使用户提到"上次去迪士尼的照片"，系统也无法找到相关记忆

## 实现方案

### 1. 数据库扩展

给 `MemoryRecord` 添加了可选的 `embedding` 字段：

```typescript
export interface MemoryRecord {
  id: string
  characterId: string
  name: string
  description: string
  type: MemoryType
  content: string
  updatedAt: number
  embedding?: number[] // 新增：向量表示
}
```

三端数据库均已更新：
- **移动端** (`stage-pocket`): SQLite + Drizzle，embedding 以 JSON 字符串存储
- **桌面端** (`stage-tamagotchi`): better-sqlite3 + IPC，embedding 以 JSON 字符串存储
- **Web 端** (`stage-web`): IndexedDB，直接存储数组

### 2. Embedding 生成（松耦合设计）

在 `memory` store 中添加了 embedding provider 配置：

```typescript
const embeddingProviderId = ref<string | null>(null)
const embeddingModel = ref<string | null>(null)
```

关键特性：
- **可选功能**：没有配置 embedding provider 时，记忆系统仍正常工作
- **失败不阻塞**：embedding 生成失败不会阻止记忆保存
- **自动生成**：保存记忆时自动生成 embedding

支持的 embedding 服务商：任何兼容 OpenAI API 的服务商（OpenAI text-embedding-3-small 推荐）。

### 3. 相关性召回

添加了 `findRelevantMemories(query, k)` 方法：

1. 获取用户的 query（最新消息）
2. 调用 embedding API 生成 query 的向量表示
3. 计算与所有记忆的余弦相似度
4. 返回 top-K 最相关的记忆

```typescript
async function findRelevantMemories(query: string, k: number = 5): Promise<MemoryRecord[]>
```

### 4. 智能注入策略

更新了 `createMemoryContext` 函数：

| 记忆数量 | Embedding 配置 | 行为 |
|---|---|---|
| < 30 条 | 任意 | 全量注入（无需检索） |
| ≥ 30 条 | 未配置 | 全量注入 |
| ≥ 30 条 | 已配置 | **Embedding 召回 top-5** |

这样既保证了小数据量的效率，又解决了大数据量的体验问题。

## 技术细节

### 余弦相似度

```typescript
function cosineSimilarity(a: number[], b: number[]): number {
  let dotProduct = 0; let normA = 0; let normB = 0
  for (let i = 0; i < a.length; i++) {
    dotProduct += a[i] * b[i]
    normA += a[i] * a[i]
    normB += b[i] * b[i]
  }
  return dotProduct / (Math.sqrt(normA) * Math.sqrt(normB))
}
```

## 使用方式

### 配置 Embedding Provider

```typescript
import { useMemoryStore } from '@proj-airi/stage-ui/stores/memory'

const memoryStore = useMemoryStore()

// 配置 OpenAI embedding
memoryStore.setEmbeddingProvider('openai', 'text-embedding-3-small')

// 配置其他兼容服务商
memoryStore.setEmbeddingProvider('custom-provider', 'embedding-model-id')
```

### 自动工作流

1. 用户发送消息
2. `createMemoryContext({ latestUserMessage })` 被调用
3. 如果记忆 ≥ 30 条且配置了 embedding provider：
   - 调用 `findRelevantMemories(latestUserMessage, 5)`
   - 只注入 top-5 相关记忆
4. 否则：全量注入

## 效果预期

| 场景 | 之前 | 之后 |
|---|---|---|
| 用户说"上次去迪士尼的照片" | 只看到索引 | 找到并显示相关记忆 |
| 100 条记忆时的对话 | 上下文 25KB 限制 | 只注入 5 条相关记忆 |
| Embedding API 失败 | 正常 | 降级为全量注入 |
| 未配置 Embedding | 正常 | 全量注入（无变化） |

## 后续计划

当前只在 session 切换时提取记忆。长期对话不切 session 时会丢失信息。计划添加：
- 每 N 条消息触发一次提取
- 前台空闲时触发提取

### 高级召回策略

当前只使用 embedding 相似度。未来可加入：
- 类型权重（feedback 类型在用户纠正时优先）
- 时间衰减（太旧的记忆降权）
- 情感互补（用户伤心时优先找正面记忆）

## 松耦合设计

所有新增功能都是可选的：
1. 没有配置 embedding provider → 系统按原方式工作
2. embedding 生成失败 → 记忆仍正常保存，只是没有向量
3. 查询失败 → 降级为全量注入
4. 数据库迁移是向后兼容的（embedding 字段可选）

这样确保了稳定性和渐进式升级的可能性。

## 成本估算

以 OpenAI text-embedding-3-small 为例：

| 操作 | 成本 | 频率 |
|---|---|---|
| 入库（每条新记忆） | $0.00002 / 1M tokens | 每次 upsertMemory |
| 查询（每次对话） | $0.00002 / 1M tokens | 每次发送消息 |

假设一个用户一年沉淀 500 条记忆，进行 1000 次对话：
- 总成本约 **$0.02-0.05 / 用户 / 年**
